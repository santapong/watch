## Executive summary

Goal: detect objects fast and put a trustworthy distance/range on each one, in real time, on consumer webcams and edge boxes (CPU / Jetson Orin), inside this repo's existing detection + depth seams.

WHAT WORKS TODAY (verified against the code and primary sources):
1) Detection is solved and already extensible. The detector factory is generation-agnostic: src/models/registry.py:39 already aliases "yolo26" to YOLODetector and _infer_family() routes any non-RT-DETR weight to it, so Ultralytics YOLO26 (NMS-free, edge-first; AGPL-3.0; official weights shipped via the ultralytics package in Jan 2026; ~40.9-57.5 mAP at 1.7-11.8 ms T4-TensorRT) is a config-only swap. NMS-free decoding removes the variable-latency NMS step, giving a steadier detect->depth->alert frame time on CPU/Jetson, and tighter small-object boxes (STAL/ProgLoss) mean cleaner in-box depth sampling. RT-DETR remains available behind the same factory for a higher-accuracy GPU option.
2) Monocular RELATIVE depth -> per-object distance is already wired end to end. src/depth/onnx_estimator.py (DepthAnythingV2 / MiDaS + build_depth_estimator) feeds the pure numpy sampler in src/depth/base.py (sample_depth: shrunk-bbox median + MAD), which populates Detection.depth (src/models/base.py:19) and drives run_depth.py proximity alerts. Depth Anything V2-Small is the only edge-viable AND commercially-licensed (Apache-2.0) variant; onnxruntime is already imported lazily, satisfying the import-safety invariant.
3) The honest gap is METERS, not plumbing. Today Detection.depth is a unitless inverse-depth score (base.py:27: "larger = nearer"), and run_depth.py:102 alerts on depth >= 0.8 (normalized). To say "X meters" you need either (a) a metric model, (b) a one-time per-camera scale calibration, or (c) calibrated geometry. The single most error-prone change for any metric path is the CONVENTION FLIP: meters mean smaller = nearer, so the metric path must bypass percentile_normalize, carry a units flag, and invert the comparison to depth <= threshold_m. This sign flip is the top correctness item and is fully unit-testable with a synthetic map + mocked session.

PRACTICAL VERDICT for an uncalibrated webcam, today: YOLO26-Small (or yolo11n) detection + Depth Anything V2-Small (Apache-2.0) relative depth, sampled with the existing median/MAD helper, plus a one-time linear scale calibration to convert relative inverse-depth to approximate meters. For a fixed, roughly-flat ground camera (traffic/retail/robot lane), calibrated single-view ground-plane geometry (WACV 2020, Ali et al.; RMSE ~6.1-7.3 m on KITTI/nuScenes/Lyft) gives true meters on pure CPU with no neural depth at all, reusing the homography primitives already in src/multicam/geometry.py. For a true-metric, far-range rig, real-time stereo (ESMStereo: 91 FPS on AGX Orin, 1.7M params, MIT, ONNX+TRT export) is the best surveyed option but needs stereo hardware.

The roadmap below is ordered to ship metric range with the least code and risk first (calibration + units on the path already in the repo), then add geometry and temporal smoothing, then gate the heavier/strings-attached options (stereo hardware, oVDA's non-commercial license) behind config and validate them on a Jetson/torch box per the repo's "test the pure core, mock the model" rule. Every item is additive, config-gated, off by default, and keeps heavy imports lazy.

### Roadmap rationale

**1. Metric depth backend + the depth-convention flip (meters end-to-end)** (P0/M)

Highest value-per-effort path to real meters. Add backend 'depth_anything_metric' to build_depth_estimator loading the Apache-2.0 Depth-Anything-V2-Metric-Hypersim-Small (indoor, ~20 m) or VKITTI-Small (outdoor, ~80 m) ONNX. The model returns meters (smaller = nearer), so this backend must NOT call percentile_normalize. Carry a units flag (relative|metric) on the estimator and Detection so consumers know the scale. THE CORE CORRECTNESS ITEM: flip run_depth.py:102 from 'depth >= 0.8 (normalized)' to 'depth <= proximity_threshold (meters)'. sample_depth (median+MAD) is unchanged. Pure logic (units handling, meters threshold, sampling on a synthetic metric map) is unit-testable torch/onnx-free with a mocked session; real forward + in-meters accuracy validated separately on a torch/TensorRT box. Onnxruntime stays lazy.

_Module:_ `src/depth/onnx_estimator.py (new backend in build_depth_estimator) + src/depth/base.py (units flag/docstring) + src/models/base.py (Detection units) + run_depth.py (meters threshold + sign flip) + configs/default.yaml depth.*`

**2. One-time relative->meters scale calibration helper (works on the existing relative DAv2/MiDaS path)** (P0/S)

Lets users get approximate meters WITHOUT swapping models, on pure CPU/Jetson. Single-image metric scale is intrinsics-dependent, so add a small calibrator: from one (or a few) known-distance references (a target at a measured range, or a known real-world object height in px<->m), fit a linear/affine map from relative inverse-depth to meters, sampled via the existing sample_depth. Pure numpy, no heavy deps, fully unit-testable. Pair with the units flag from the metric item so run_depth.py thresholds can be specified in meters regardless of backend. This is the cheapest route to a meters-denominated proximity alert.

_Module:_ `new src/depth/calibration.py (pure helper) wired into build_depth_estimator/run_depth.py + configs/default.yaml depth.scale`

**3. Edge input-size knob for the depth backend (FPS/accuracy trade without code changes)** (P0/S)

build_depth_estimator already reads input_size; the default 518x518 is slow on Jetson (community reports ~10 FPS class on Orin). Document and expose a lower input_size (~308-406) in the depth config block for a roughly 2-4x speedup, defaults unchanged. Zero new code beyond config/docs; keeps the detect->depth->alert loop real-time on edge. Validate the actual FPS on the target device before committing numbers to the roadmap (current edge figures are from guides/blogs, not measured in-repo).

_Module:_ `configs/default.yaml depth.input_size + docs note in run_depth.py`

**4. Per-track distance Kalman filter (smoothed range + closing speed, fills gaps)** (P0/S)

TrackInfo today stores only pixel positions and distance_traveled in PIXELS (src/tracking/tracker.py:38-47) with no range or velocity. Add a small 1D constant-velocity numpy KF over Detection.depth keyed by track_id (Detection.track_id, base.py:18) exposing smoothed_depth + range-rate (dZ/dt) and predicting through detection gaps. This denoises per-frame depth jitter into stable range/speed and is the prerequisite signal for TTC. Pure numpy, license-free, GPU-free, mocks cleanly with synthetic tracks. High ROI, low effort, changes no existing behavior.

_Module:_ `src/tracking/tracker.py (extend TrackInfo) or a new src/tracking/range_filter.py`

**5. YOLO26 as a config-selectable, benchmarked detector option** (P1/S)

Already routable: registry.py:39 aliases 'yolo26' to YOLODetector and the wrapper/tracking/export paths are generation-agnostic, so build_detector({'name':'yolo26n.pt'}) works with no new code. NMS-free inference removes the variable-latency NMS post-step for a steadier real-time cadence on CPU/Jetson, and tighter small-object boxes improve in-box depth sampling for Detection.depth. Ship a config entry + a documented edge benchmark + a unit test asserting the alias/loader (model mocked per golden rules). Risks: AGPL-3.0; requires an ultralytics version new enough to load YOLO26 (pin requirements.txt; keep ultralytics stubbed in tests/conftest.py so CI stays torch-free); end-to-end head emits fixed top-k boxes so iou_threshold becomes a no-op (harmless) - confirm downstream tracking/zone logic assumes no NMS step. Validate edge FPS on the actual device.

_Module:_ `configs/default.yaml model.name + tests/ alias-loader test + requirements.txt pin`

**6. Single-view ground-plane ranger (true meters on CPU for fixed, roughly-flat-ground cameras)** (P1/M)

For a mounted traffic/retail/robot camera this gives absolute meters with NO neural depth, reusing primitives already in src/multicam/geometry.py: build_homography (image<->ground correspondences), ground_point (bbox foot point), project_point (back-projection). Add a GroundPlaneRanger implementing the BaseDepthEstimator-style output that takes a calibrated homography + camera height/intrinsics and writes metric Detection.depth, so run_depth.py works unchanged but in true meters. Per WACV 2020 (Ali et al.) RMSE ~6.1-7.3 m on KITTI/nuScenes/Lyft. Core math is pure numpy/cv2 (slim-deps-safe, fully unit-testable). Risks: needs per-camera calibration (known ground reference or lane width); sensitive to pitch/roll and non-flat ground - smooth the foot point with the existing tracker. Gate behind depth.backend: 'ground_plane'.

_Module:_ `new src/depth/ground_plane.py (or src/geometry/) reusing src/multicam/geometry.py + build_depth_estimator + configs/default.yaml`

**7. Time-to-collision / looming estimator (tracker-driven, depth-optional)** (P1/S)

Closed-form TTC from bounding-box scale-change (and the FOE/optical-flow form for off-axis targets), consuming per-track box-size + timestamp history that the tracker/temporal buffer already keep (temporal.py already computes a px/frame speed anomaly, so a closing-rate/TTC field is a natural sibling). Emit a 'collision' alert via the existing AlertManager/AlertRule, mirroring the proximity rule in run_depth.py, behind a --ttc flag or scripts/run_ttc.py. When depth is present, use the TTC = -Z/(dZ/dt) form off the per-track range filter so it works with or without the depth subsystem. Pure numpy, no heavy deps, off by default. Risks: box noise/oscillation needs smoothing + a deadband (themselves unit-tested); absolute-range claims still defer to depth. A learned upgrade (e.g. FP-TTC) can later sit behind a build_* factory without changing this head.

_Module:_ `new src/analytics/ttc.py + AlertRule in run_depth.py / new scripts/run_ttc.py`

**8. Real-time stereo subsystem for true far-range metric depth (ESMStereo)** (P2/L)

Best surveyed real-time/edge stereo: ESMStereo (arXiv 2506.21091, MIT) reports 91 FPS on AGX Orin / 116 FPS high-end GPU at 1.7M params, with ONNX export + a trtexec FP16 command. Add a new src/stereo/ StereoMatcher mirroring BaseDepthEstimator with an ONNX/TRT backend reusing the OnnxDepthEstimator pattern, a cv2.stereoRectify/initUndistortRectifyMap helper near src/multicam/geometry.py, and a disparity->depth converter Z=f*B/d. Crucially sample_depth/annotate_depth work UNCHANGED on a stereo map, so the detect->range plumbing and run_depth.py alerts are already done - only the map SOURCE changes; detection stays YOLO/RT-DETR. Gate behind the depth/stereo factory, off by default. Risks: requires a stereo rig (chief blocker); KITTI/Scene-Flow weights may need fine-tuning for a new baseline; far-field accuracy is baseline-limited; validate the forward on a GPU/Jetson box, not CI. Unit-test only the pure parts (disparity->depth, rectification-map application, loader/factory).

_Module:_ `new src/stereo/ (StereoMatcher + ONNX/TRT backend) + rectification helper near src/multicam/geometry.py + factory wiring`

**9. Temporally-consistent streaming depth backend (oVDA) as an opt-in GPU/Jetson option** (P2/L)

Online Video Depth Anything (arXiv 2510.09182) caches latent features across frames for flicker-free streaming depth; reported 42 FPS on A100, 20 FPS on a Jetson edge device. Insertion is clean: a STATEFUL estimator implementing BaseDepthEstimator, registered as backend 'ovda', that carries cache state across estimate() calls with a reset() on stream open/seek - the one real design change vs today's stateless estimators. Output stays an HxW relative-depth map, so annotate_depth/sample_depth/Detection.depth/run_depth.py need no change and proximity-alert flicker drops out for free. Risks (why it is P2/last): non-commercial license is the biggest gate; streaming ONNX/TRT export for a cached-state model is unproven in-repo; ViT-S oVDA is likely sub-real-time on pure CPU (real-time path is Jetson Orin FP16 or a desktop GPU); output is still relative depth so metric range needs the scale calibration above. Recommend shipping the per-track range KF first, then piloting oVDA only after the license is cleared.

_Module:_ `src/depth/onnx_estimator.py (new stateful 'ovda' backend in build_depth_estimator) + src/depth/base.py (estimator lifecycle/reset)`

## Findings by theme

## Real-time object detection

For this platform the detector is the front half of a two-stage *detect -> range* loop: it must localize objects fast enough to leave a frame-time budget for the monocular-depth subsystem (`src/depth/`, ONNX Depth-Anything/MiDaS) that samples pixels inside each box to populate `Detection.depth`, which `scripts/run_depth.py` turns into proximity alerts. So the detector is judged on three axes that matter more here than raw COCO mAP: (1) stable, low frame-time on consumer webcams and edge (CPU/Jetson) so depth has headroom; (2) tight, high-recall boxes — especially on small objects — because depth is read from pixels *inside* the box, so a loose or missed box yields a noisy or absent range estimate; and (3) clean quantization/export for edge hardware. The synthesis below is drawn from two verified research cards covering a YOLO26-based detector and a YOLO26 + Depth-Anything-V2 edge proximity recipe.

#### Method: Ultralytics YOLO26 (NMS-free end-to-end detector)

**One-line idea.** An anchor-free, single-stage YOLO that is *natively end-to-end* — it removes Non-Maximum Suppression (NMS) from inference via a one-to-one (top-1 matching) head, and removes Distribution Focal Loss (DFL) from the regression head, for a simpler graph, cleaner ONNX/TensorRT/INT8 export, and faster CPU post-processing [[Jocher/Qiu/Liu et al., 2026](https://arxiv.org/abs/2606.03748)].

**How it works.** YOLO26 trains a shared backbone+neck with a dual-head scheme: a classic one-to-many head feeds dense gradients during training, while a one-to-one head learns to emit exactly one box per object so it can be used *alone* at inference, with no NMS threshold to tune and no variable per-frame NMS latency — the dominant CPU/edge win. Removing DFL replaces distributional box decoding with lighter direct regression, which is the key enabler for clean INT8 because DFL's custom decode op historically broke or slowed integer quantization. Training-side additions (no inference cost) — STAL (Small-Target-Aware Label Assignment), ProgLoss (progressive supervision), and the MuSGD optimizer — specifically raise small-object recall and stabilize training [[Jocher/Qiu/Liu et al., 2026](https://arxiv.org/abs/2606.03748); [Ultralytics Docs, 2026](https://docs.ultralytics.com/models/yolo26)].

**Accuracy (COCO val2017 mAP50-95 @640px).** Per the official Ultralytics model docs and benchmark macro:

| Variant | mAP (with-NMS) | mAP (end-to-end) | Params | FLOPs |
|---|---|---|---|---|
| YOLO26n | 40.9 | — | 2.4 M | 5.4 B |
| YOLO26s | 48.6 | 47.8 | 9.5 M | 20.7 B |
| YOLO26m | 53.1 | 52.5 | 20.4 M | 68.2 B |
| YOLO26l | 55.0 | 54.4 | 24.8 M | 86.4 B |
| YOLO26x | 57.5 | 56.9 | 55.7 M | 193.9 B |

The nano variant's 40.9 mAP is a notable jump over YOLOv8n's 37.3 at fewer params/FLOPs [[Ultralytics Docs, 2026](https://docs.ultralytics.com/models/yolo26)]. (The end-to-end column reflects the documented ~0.6–0.8 AP gap between the with-NMS and one-to-one heads; those exact cells are consistent with the official gap but were not digit-confirmed for the nano variant.)

**Latency / FPS (with the hardware noted).** All figures from the official benchmark table unless flagged:

| Hardware / runtime | n | s | m | l | x |
|---|---|---|---|---|---|
| **GPU** — NVIDIA T4, TensorRT 10 FP16 (timed over COCO val on EC2 P4d) | 1.7 ms | 2.5 ms | 4.7 ms | 6.2 ms | 11.8 ms |
| **CPU** — Intel Xeon @2.00 GHz, ONNX Runtime | 38.9 ms | 87.2 ms | 220.0 ms | 286.2 ms | 525.8 ms |

The headline CPU result: YOLO26n reaches 40.9 mAP at 38.9 ms on CPU vs YOLOv8n's 37.3 mAP at 80.4 ms (Intel Xeon @2.00 GHz) — higher accuracy *and* ~2x faster, driven by NMS + DFL removal; the "up to 43% faster CPU" figure is measured vs YOLO11n/YOLOv12n, not v8n [[Ultralytics Docs (YOLO26-vs-YOLOv8), 2026](https://docs.ultralytics.com/compare/yolo26-vs-yolov8); [Jocher/Qiu/Liu et al., 2026](https://arxiv.org/abs/2606.03748)]. Validated Jetson numbers (JetPack 6.1, Ultralytics 8.4.33) confirm strong edge throughput:

| Board | YOLO26n FP16 | YOLO26n INT8 | YOLO26s FP16 | YOLO26s INT8 |
|---|---|---|---|---|
| Orin Nano Super Dev Kit | 4.57 ms (mAP 0.480) | 3.80 ms (mAP 0.449) | 7.17 ms (0.565) | 5.25 ms (0.547) |
| Orin NX 16 GB | 4.13 ms | 3.49 ms | 6.41 ms | 4.78 ms |

INT8 costs ~3 mAP (0.479 -> 0.449 for n) for ~263 FPS on Orin Nano [[Ultralytics Jetson benchmarks, 2025](https://docs.ultralytics.com/models/yolo26)]. Raspberry Pi 5 / lower-tier Jetson FPS (~7–8 FPS @640 on Pi 5; ~6–10 FPS on the original Nano) come from Ultralytics guides/third-party posts, **not** the peer-reviewed table — treat as indicative and re-measure on the target device.

**Maturity.** GA January 2026. Official Ultralytics weights (n/s/m/l/x), docs, and Jetson/DeepStream/TensorRT export guides; the same checkpoints also do segmentation, pose, OBB, classification, and open-vocabulary (YOLOE-26). License: **AGPL-3.0** for code and weights, with a paid Enterprise license required for closed-source/commercial distribution — a genuine gate if this repo is ever shipped commercially (the permissively-licensed RT-DETRv2 / D-FINE families are the Apache-2.0 fallbacks).

**Fit for this repo.** Essentially drop-in for `src/models/`. Verified in the local code: `src/models/registry.py:39` already lists `"yolo26"` in the alias tuple registered to `YOLODetector`, and `_infer_family()` returns `"yolo"` for any non-RT-DETR name, so a `yolo26n.pt` weight routes through `build_detector` with **no new wrapper, tracking, or export code**. INT8/FP16/engine/OpenVINO export is already wired in `src/deployment/exporter.py`. Effort: low — a config entry (`configs/default.yaml` model name = `yolo26n.pt`), a version pin of `ultralytics`, and a unit test asserting the alias/loader with the model mocked (per the repo's "test the pure core, mock the model" rule). The NMS-free head makes `iou_threshold` a harmless no-op. **Risks:** (a) AGPL-3.0; (b) requires an `ultralytics` new enough to load YOLO26 weights, so keep `ultralytics` stubbed in `tests/conftest.py` so CI stays torch-free; (c) confirm downstream tracking/zone logic does not assume an NMS step; (d) edge FPS on Pi/Jetson is guide-level — validate on-device before committing roadmap numbers; (e) on-webcam *range* accuracy depends on per-class box tightness, not COCO mAP, so validate against the depth pipeline on-device.

#### Pairing it for range: YOLO26 + Depth Anything V2-Small (metric)

The second card specifies the end-to-end edge proximity recipe that directly serves the distance/range mission: **YOLO26 INT8** (Jetson) or INT8 ONNX/OpenVINO (CPU) for detection, paired with **Depth Anything V2-Small** (ViT-S + DPT, 24.8 M params) for per-object range [[Yang et al., 2024](https://arxiv.org/abs/2406.09414)]. The base model outputs *relative* inverse depth; the official **metric** fine-tunes output absolute meters directly — indoor on Hypersim (max 20 m), outdoor on Virtual KITTI 2 (max 80 m) [[Depth Anything V2 metric_depth, 2024](https://github.com/DepthAnything/Depth-Anything-V2/blob/main/metric_depth/README.md)]. Per-object range = sample the depth map inside each box and take a robust statistic, exactly what the repo's `src/depth/base.py:48` `sample_depth()` already does (20% bbox shrink + 3-sigma MAD + median).

**Key deployment finding — INT8 belongs on the detector, NOT depth.** TensorRT INT8 on the ViT+DPT depth architecture *regresses* speed because it breaks the fused FP16 attention kernels into plain matmuls: an NVIDIA-forum benchmark of MiDaS V3 DPT-Hybrid on AGX Thor shows FP16 5.8 ms vs INT8 10.3 ms (~1.8x **slower**). So keep Depth-Anything at FP16/BF16.

**The bottleneck is depth, not detection.** Indicative ViT-S+DPT depth latency on a Jetson Orin Dev Kit 8 GB (TensorRT) scales sharply with input size: ~23.5 ms @308px (~42 FPS), ~47.7 ms @406px (~21 FPS), ~98.0 ms @518px (~10 FPS, the repo default), VRAM 626–689 MB; Base/Large do not fit in 8 GB. **Caveat (flagged in verification):** this latency table is from the IRCVLab *Depth-Anything-for-Jetson-Orin* benchmark, which runs Depth Anything **V1** (ViT-S14 + DPT, same 24.8 M architecture) and does not state FP16 — so treat it as a same-architecture proxy for V2-Small, not a confirmed V2-FP16 measurement; re-measure on the target board. A sequential joint budget on Orin Nano is roughly YOLO26n INT8 (~3.8 ms) + DAV2-S FP16 @406px (~47.7 ms) ≈ **~52 ms/frame (~19 FPS)** — depth dominates. Two cheap wins, both already supported by `build_depth_estimator`'s `input_size`: lower depth resolution 518 -> 384/406 (2–4x faster, no model change), and run depth every Nth frame, reusing the last map for tracked boxes.

**Depth licensing aligns with the constraint.** DAV2-**Small** is Apache-2.0 (commercial OK) — Base/Large/Giant are CC-BY-NC-4.0 (non-commercial), and the metric checkpoints follow the same per-size split. Small is therefore both the only commercially-clean metric tier *and* the only one that fits 8 GB Jetson.

**Fit for this repo (depth side).** Two low-effort changes: (1) lower `input_size` from 518 to ~384–406 in the depth config; (2) add a metric backend (e.g. `backend='depth_anything_metric'`) so `Detection.depth` is **meters** and skip `percentile_normalize` for it. The one correctness item to get right and unit-test: the **sign convention flips** — the repo currently treats `Detection.depth` as inverse depth (larger = nearer, `proximity_threshold=0.8` in `run_depth.py`), but metric meters invert that (smaller = nearer), so the proximity rule and its tests must change together or alerts fire backwards. The in-meters error of the metric model is *not* paper-verifiable and is scale-fragile out-of-domain — a one-time per-camera scale check against a known distance is advisable; leave it as an on-platform calibration task.

#### Real-time verdict and ranked recommendation

| Method | Real-time on CPU webcam? | Real-time on edge (Jetson)? | Best role here |
|---|---|---|---|
| YOLO26n (INT8/ONNX) | Yes — 38.9 ms (~26 FPS) on Xeon CPU | Yes — 3.5–4.6 ms on Orin Nano/NX | Default detector for the detect->range loop |
| YOLO26s | Marginal — 87 ms CPU (often run @416px) | Yes — 4.8–7.2 ms on Orin | Accuracy bump when a GPU/Jetson is present |
| YOLO26 m/l/x | No — 220–526 ms CPU | GPU/AGX-class only | Offline/server detection, not edge |
| Depth Anything V2-Small (FP16) | Constrained | ~10–42 FPS depending on input res; depth dominates the budget | Per-object metric range (keep FP16, lower input size) |

**Ranked roadmap recommendations:**

1. **Adopt YOLO26n as the default real-time detector (highest priority, lowest effort).** It is the only variant that is comfortably real-time on a CPU-only webcam (~26 FPS) *and* extremely fast on Jetson (~263 FPS INT8), at higher accuracy than the current YOLOv8n path. The NMS-free + DFL-free design removes the variable-latency NMS post-step, giving the `detect->depth->alert` loop a steadier cadence, and STAL/ProgLoss tighten small-object boxes for cleaner in-box depth sampling. It already routes through `build_detector` (`registry.py:39`), so the work is a config entry + version pin + a mocked loader test. *Why:* maximum payoff for the range mission at near-zero integration cost.
2. **Ship the metric-depth backend + sign-convention fix (high priority, ~1 small PR).** This is what converts the existing relative-depth proximity heuristic into a true "closer than X meters" alert. The readout (`sample_depth`) and export paths already exist; the load-bearing work is the meters threshold + the inverse->metric sign flip in `run_depth.py`, which is pure, mockable logic that satisfies the repo's testing rule. *Why:* it is the single change that makes the platform's distance output physically meaningful and commercially-licensed (Apache-2.0 Small).

4. **Validate on-device before quoting numbers, and watch licensing (ongoing).** Re-measure Pi/Jetson detector FPS and the V2-Small depth latency on the actual target board (the cited depth latency is a V1 proxy), verify the INT8 TensorRT engine build against your JetPack, run a per-camera metric-depth scale check, and decide whether AGPL-3.0 (YOLO26) is acceptable or whether an Apache-2.0 detector (RT-DETRv2 / D-FINE) is needed for commercial distribution.

## Distance & depth estimation methods

This section synthesizes four method families for estimating **metric range to detected objects** in real time on consumer webcams and edge hardware (CPU / Jetson). The target platform already ships the plumbing: a 2D detector behind a factory (YOLO / RT-DETR), a monocular depth subsystem (`src/depth/`, ONNX Depth-Anything / MiDaS) that writes `Detection.depth`, the `scripts/run_depth.py` proximity-alert entrypoint, and multicam geometry (homography + Hungarian) in `src/multicam/geometry.py`. The central gap is that today `Detection.depth` carries **relative inverse depth in [0, 1]** (re-scaled per frame by `percentile_normalize`), not metres — so "distance" is currently unitless and frame-relative. The methods below are evaluated as ways to close that gap.

The four families trade off along one axis: **does the method need extra hardware or per-camera calibration to produce trustworthy metres?**

- **Monocular metric depth** (Depth Anything V2 metric) — single RGB camera, but metric scale is intrinsics-dependent, so a one-time per-camera calibration is effectively required.
- **Calibrated stereo** (ESMStereo) — true triangulated metres, but needs a synced second camera plus rectification.
- **Single-view ground-plane geometry** (Ali et al., WACV 2020) — true metres from one camera with no depth net, but assumes a flat ground plane and a known metric scale prior (e.g. lane width).
- **Detector + depth fusion** (Dist-YOLO pattern) — not a new range primitive but the *architecture this repo already runs*; the roadmap value is upgrading its depth source from relative to metric.

#### Accuracy and the metric-scale problem

The single most important constraint for "range to object in metres" is that **monocular metric scale is intrinsics-dependent**. Depth Anything V2's metric checkpoints are two *separate* heads — indoor (Hypersim, capped 20 m) and outdoor (Virtual KITTI 2, capped 80 m) — each regressing absolute metres ([DepthAnything, 2024](https://github.com/DepthAnything/Depth-Anything-V2/blob/main/metric_depth/README.md)). The headline NYU-D AbsRel of **0.043 belongs to the Large variant** (the Small number is lower and was not readable from a primary card), and independent users reproducing the metric models on KITTI report δ1 ≈ 0.8xx and RMSE ≈ 4.x versus the paper's ~3.x ([repo issue #88](https://github.com/DepthAnything/Depth-Anything-V2/issues/88)) — i.e. **uncalibrated real-world numbers degrade noticeably**. The practical implication for an uncalibrated webcam is that a **one-time per-camera linear scale calibration** (a known-distance reference) is effectively required for trustworthy metres.

By contrast, **calibrated stereo gives true triangulated metres** and is the only family here that is metric *by construction* — but it is bounded by the stereo error law (depth error grows ∼ Z²/(f·B)), so a webcam-scale baseline limits reliable range to the near/mid field, and the fastest ESMStereo-S config is also the least accurate ([Tahmasebi et al., 2025](https://arxiv.org/abs/2506.21091)).

The **ground-plane method** also yields true metres (RMSE 6.10–7.31 m at automotive ranges) and is notably **robust across cameras without retraining** — the authors show a learned monocular baseline degrades to RMSE 12.23 m on a different camera while the geometric method does not ([Ali et al., 2020](https://openaccess.thecvf.com/content_WACV_2020/papers/Ali_Real-time_vehicle_distance_estimation_using_single_view_geometry_WACV_2020_paper.pdf)). Its accuracy is coarser than learned metric depth, so it suits **proximity ranking / alerts** more than centimetre-grade measurement, and it assumes a flat ground plane plus visible lane markings (a known scale prior), which limits it to road-like / floor-standing scenes.

#### Fit-for-repo notes

- **Depth Anything V2 metric** is a near drop-in for `src/depth/`. The repo's `preprocess()` in `src/depth/onnx_estimator.py` already does exactly the 518×518 + ImageNet normalization + NCHW these checkpoints need, and `build_depth_estimator()` already dispatches `depth_anything`/`dav2`/`midas` backends. Effort is **LOW–MEDIUM**: (1) add a metric ONNX backend (e.g. `depth_anything_metric`) with an `is_metric`/`max_depth` flag, gated via config and off by default per the additive rule; (2) **handle the convention flip** — `src/depth/base.py` documents "larger = nearer (inverse depth)" and `percentile_normalize` returns a unitless [0,1] map, so the metric path must bypass `percentile_normalize`, pass metres straight through (smaller = nearer), and write `Detection.depth` in metres; (3) flip and re-unit the proximity comparison in `scripts/run_depth.py` (it currently thresholds the normalized value at 0.8 with `det.depth >= proximity`). The pure helper `sample_depth()` (median of a shrunk bbox + 3-sigma MAD) works unchanged and is ideal for range-to-object. Tests stay torch/ONNX-free by mocking the session, exactly as the current depth tests do.

- **ESMStereo** is the strongest real-time/edge stereo option but needs new hardware. Recommended shape: a new `src/stereo/` subsystem with a `StereoMatcher` interface mirroring `BaseDepthEstimator`, an ONNX/TensorRT backend reusing the `OnnxDepthEstimator` pattern, a rectification helper living near `src/multicam/geometry.py`, and a small `Z = f·B/d` converter. Crucially, `sample_depth()`/`annotate_depth()` work **unchanged** on a stereo depth map, so the detection→range plumbing and proximity alerts are already done — only the map *source* changes. Effort **MODERATE**, bounded by one-time calibration utilities and TensorRT engine handling; the chief blocker is requiring a real synced stereo rig.

- **Ground-plane geometry** is the best webcam/edge match in the no-extra-hardware family and maps almost 1:1 onto existing primitives: `build_homography()`, `ground_point()` (already returns the bbox bottom-centre foot point), and `project_point()` in `src/multicam/geometry.py`. Recommended shape: a `GroundPlaneRanger` (e.g. `src/depth/ground_plane.py`) implementing `BaseDepthEstimator`-style output that writes **absolute metres** into `Detection.depth`, gated behind a `depth.backend: "ground_plane"` config option. Effort **LOW** for the core (pure numpy/cv2, no torch — fits the slim-deps invariant), **MODERATE** if the lane-cross-ratio horizon recovery is automated rather than statically calibrated. **Risk:** the underlying technique is patented by Motive (granted US patents incl. US 11,798,187 and US 12,175,695) — the textbook projective primitives are free to use, but a faithful reimplementation of the specific WACV method needs IP review before commercial deployment.

- **Detector + depth fusion** is not a new architecture to build — it is the one `src/depth/` already ships (`run_depth.py` runs `detector.detect`, then `depth_estimator.estimate`, `percentile_normalize`, `annotate_depth`→`sample_depth`). The roadmap value is **quality**: swap the relative checkpoint for a metric one and a calibrated relative→metres mapping so the proximity threshold can be specified in metres. The full Dist-YOLO single-head route (a trained distance head on the YOLO wrapper) is **higher risk** — it needs KITTI-style distance ground truth and an effectively fixed camera, and does not transfer to an arbitrary webcam without retraining — so it is the lower priority.

#### Ranked recommendation for roadmap work

1. **Add a metric Depth Anything V2-Small ONNX backend + convention-flip + metres-based proximity (LOW–MEDIUM, do first).** This delivers the headline outcome — `Detection.depth` in true metres — with the smallest change, on the single-camera hardware users already have, and is fully additive/config-gated. It directly upgrades the existing fusion pipeline rather than introducing a new subsystem. **Why first:** highest value-to-effort, reuses `preprocess`/`sample_depth`/`run_depth.py` as-is, Apache-2.0 Small weights are commercially clean, and it is testable torch-free with a mocked session. 2. **Add a `GroundPlaneRanger` single-view geometric backend (LOW core, do second — gated on IP review).** Best fit to the existing `src/multicam/geometry.py` primitives, **CPU real-time** (the only family with a confirmed CPU figure), and cross-camera robust without retraining. Ideal as a lightweight, calibration-driven metric option for floor/road scenes and as a sanity cross-check against the learned depth path. **Caveat:** flat-ground assumption + the Motive patents — scope it for non-commercial / internal use, or clear IP, before shipping.

3. **Scope (don't build yet) an `src/stereo/` ESMStereo backend (MODERATE, hardware-gated).** Technically the most accurate and genuinely real-time-on-edge route to metric depth (MIT-licensed, ~91 FPS S-gwc on AGX Orin), and it reuses the depth interface and `sample_depth` cleanly. **Why later:** it requires a synced, rectified stereo rig that the current single-webcam platform does not assume, and far-field accuracy is baseline-limited. Worth a roadmap entry as the high-accuracy upgrade once stereo hardware is in scope.

4. **De-prioritize the full Dist-YOLO single-head route.** The cooperating-models pattern it formalizes is already realized in this repo; a trained distance head adds KITTI-style training and a fixed-camera assumption for marginal benefit over option 1. Keep it as a reference for the fusion design, not as build work.

## Attaching distance to detections: fusion, collision & tracking

This section covers how to attach a **range/distance** estimate to each detected object in real time, and how to turn that range into actionable signals (proximity, closing speed, time-to-collision) on consumer webcams and edge hardware. The platform already implements the core of the first family end-to-end — a monocular depth subsystem (`src/depth/`, ONNX Depth-Anything/MiDaS) that writes a robust per-detection scalar into `Detection.depth` via `sample_depth()` (shrink-to-center bbox + 3-sigma MAD + median), surfaced through `scripts/run_depth.py` proximity alerts. The methods below are organized as three complementary layers: (1) the **depth source** that produces the range, (2) **temporal stabilization + per-track velocity** on that range, and (3) a **collision/TTC** layer that needs no calibration at all.

A recurring caveat applies to every depth-based option here: the default output is **relative (scale-and-shift-invariant inverse) depth**, i.e. an *ordering score* where "larger = nearer", not meters. This is exactly the convention encoded in `src/depth/base.py` and is sufficient for proximity ranking and nearest-object alerts (`run_depth.py` thresholds on it), but absolute distance requires a metric checkpoint or a one-time scale calibration.

### How the layers compare and combine

These are **not competitors** so much as a stack. The depth source (DA-V2 or oVDA) produces the per-frame range; the distance Kalman filter stabilizes it across frames and adds per-object closing speed; the TTC layer is an independent, calibration-free path to collision risk that can run with *or without* the depth subsystem.

**Depth source — DA-V2 vs oVDA.** Single-frame DA-V2 ViT-S is the pragmatic default: it is already wired into the repo, the ViT-S relative checkpoint is **Apache-2.0 (commercial-safe)**, and it is the only variant that fits and runs in real time on a Jetson Orin 8GB (308–364px gets ~25–43 FPS for the depth net before detection) [[IRCVLab]](https://github.com/IRCVLab/Depth-Anything-for-Jetson-Orin). Its weakness is **per-frame flicker**: independent inference each frame makes `Detection.depth` jittery, which in turn makes any naive velocity/TTC derived from it noisy. oVDA addresses exactly this at the source — KV-caching past latents yields temporally consistent depth, so proximity-alert flicker drops out for free [[Feiden et al., 2026]](https://arxiv.org/abs/2510.09182). The trade-offs are real and roadmap-blocking: oVDA's **weights are non-commercial (NC-SA-UHD v1.0)**, it is **stateful** (the estimator must persist across frames and `reset()` on cuts/seeks — a genuine design change vs today's stateless `OnnxDepthEstimator.estimate()`), and it is realistically real-time only on **Jetson Orin NX FP16 (~20 FPS)** or a desktop GPU, not pure CPU.

**Stabilization — the distance Kalman filter.** Regardless of depth source, a tiny 1D constant-velocity KF over each track's scalar `Detection.depth` is the highest value-per-effort addition: it smooths range, gives a free closing-speed estimate (sign = approaching/receding), and predicts through detection dropouts. Note the repo's existing trackers Kalman-filter the *bounding box*, never the *distance*, and `TrackInfo.distance_traveled` is measured in **pixels** — so range/velocity is a genuinely new signal. This is pure numpy, CPU-trivial, license-free, and fully unit-testable with synthetic tracks of known range trajectories.

**Collision/TTC — the calibration-free path.** The bbox scale-change estimator is the cheapest and most robust-to-miscalibration way to get a *collision* signal, because the unknown true object size cancels in the ratio — no metric depth needed [[Manglik et al., 2019]](https://github.com/aashi7/NearCollision). It is "effectively free" relative to detection. Its Achilles' heel is **box-noise amplification**: TTC = S/(S−1) blows up as the scale ratio S→1, so YOLO box jitter produces huge TTC variance — the dominant failure mode the NearCollision paper itself flags ("small mistakes in depth estimation or noise in 2D bounding box detection can be misinterpreted as large changes in velocity") and which mandates EMA/Kalman smoothing on size plus a minimum-scale-change deadband. The closed form also assumes constant relative velocity (biased by braking/accelerating targets) and on-axis approach (off-axis objects need the optical-flow focus-of-expansion / divergence form, whose per-frame flow cost can dominate the budget on edge hardware). Crucially, **TTC is time-to-contact, not range** — absolute distance still defers to the depth subsystem. Where the depth subsystem *is* available, the same TTC head can instead be evaluated from the time-derivative of `Detection.depth` (TTC = −Z/(dZ/dt)), so one estimator serves both configurations.

### What is actually real-time on webcam/edge

- **CPU-only consumer webcam:** the **TTC bbox-scale head** and the **distance KF** are the only depth-distance signals that are comfortably real-time, because both are a handful of arithmetic ops on top of whatever the detector already costs. Single-frame DA-V2 ViT-S at 518px is marginal on CPU (reduce input size); oVDA is likely sub-real-time on CPU.
- **Jetson (Orin / Orin NX), TensorRT/FP16:** DA-V2 ViT-S is real-time for the *depth net* at reduced input (308–364px ≈ 25–43 FPS [[IRCVLab]](https://github.com/IRCVLab/Depth-Anything-for-Jetson-Orin)) — but the detector latency stacks on top, so budget both together. oVDA reaches **~20 FPS FP16 on Orin NX** [[Feiden et al., 2026]](https://arxiv.org/abs/2510.09182). DA-V2 Base/Large reportedly do **not** fit on Orin 8GB.
- **Desktop GPU (A100-class):** all options are comfortably real-time; oVDA hits 42 FPS, DA-V2 small <10 ms.
- **Latency stacks.** Every depth option's per-frame cost is *additive* to the detector. At 518px, DA-V2 ViT-S is only ~10 FPS on Orin 8GB *before* detection — so edge real-time generally needs a smaller input and/or frame-skipping the depth net. The KF and bbox-TTC layers add no meaningful latency.

### Accuracy caveats worth flagging for the roadmap

- **Relative ≠ metric.** Do not advertise `Detection.depth` as meters in default mode. DA-V2 offers per-domain **metric** checkpoints (indoor Hypersim-Small, max 20 m; outdoor VKITTI-Small, max 80 m), but they are domain-bound and do not self-predict camera intrinsics, so absolute scale can be off across very different cameras/FoVs.
- **ViT-S metric accuracy is unverified.** The well-known DA-V2 zero-shot figures (e.g. KITTI AbsRel ~0.036) belong to the **Large** checkpoint, **not** ViT-S; the cards deliberately did not attribute them to Small. Per-variant ViT-S AbsRel/δ1 should be measured in-repo rather than quoted.
- **oVDA's headline numbers measure consistency, not metric accuracy** (they are first-frame-aligned), and the exact table figures were egress-blocked during research and should be reconfirmed against the PDF (independent search corroborates the 9/20 FPS and KITTI 0.140 figures).
- **The sampler inherits map errors.** Thin/transparent/reflective objects, heavy occlusion, and curved silhouettes can still leak background depth even after shrink+MAD; per-object accuracy is best for mid-range, unoccluded objects and worst for tiny/far boxes (MAD needs ≥4 interior pixels to engage).

### Ranked recommendation for roadmap work

1. **Per-track distance Kalman filter (LOW effort, do first).** Highest value-per-effort, zero new dependencies, license-free, GPU-free. Extend `TrackInfo` in `src/tracking/tracker.py` with a 1D constant-velocity KF over `Detection.depth` keyed by `track_id`, exposing `smoothed_depth` + `speed` and predicting through gaps. It immediately denoises the *existing* depth signal and unlocks closing-speed — and it is the prerequisite that makes any depth-derived TTC usable. Pure-numpy, mocks cleanly, fully unit-testable per the repo's "test the pure core" rule.
2. **Closed-form TTC module (LOW effort, do second).** Add `src/analytics/ttc.py` (a `TTCEstimator` consuming per-track bbox/size + timestamps from `EnhancedTracker`/`TemporalBuffer`, which already keep timestamped per-track state and already compute a px/frame `SPEED_ANOMALY`). Emit a looming flag into the existing `AlertManager` via an `AlertRule(condition=lambda ctx: ctx['min_ttc'] < T, alert_type='collision', ...)`, mirroring the proximity rule in `run_depth.py`, wired by a `--ttc` flag or `scripts/run_ttc.py`. It is real-time everywhere and works with or without the depth subsystem (reuse `Detection.depth` for the −Z/(dZ/dt) form). **Must ship with the EMA/Kalman smoothing + deadband** that the box-noise failure mode requires — and those guards must themselves be unit-tested.
3. **Pin/ship DA-V2 ViT-S relative ONNX as the default depth backend + an edge input-size knob (TINY effort).** It is already wired (`DepthAnythingV2` + `build_depth_estimator` → `sample_depth`/`annotate_depth`). Make Apache-2.0 ViT-S the documented default, expose an edge input-size config (308–518) in `configs/default.yaml` so Jetson users trade FPS for accuracy without code changes, and add an optional "metric" mode (Hypersim-Small / VKITTI-Small behind the factory + a units flag on `Detection`/consumers). A one-time scale-calibration helper (known object height, or a single reference distance) is the cheapest path to approximate meters without swapping models.
4. **oVDA as an opt-in GPU/Jetson backend (MEDIUM effort, gated).** Strong technical fit (flicker-free depth → cleaner range/velocity/TTC) and it slots behind `build_depth_estimator` as a new backend key with no change to `sample_depth`/`Detection.depth`/`run_depth.py`. **Two gates hold it below the cheap wins:** (a) the **non-commercial weight license** blocks any commercial build until resolved with Heidelberg; (b) it introduces statefulness (persistent cache + `reset()` on stream change) and streaming ONNX/TRT export for a cached-state model is unproven in-repo, and it is real-time only on GPU/Jetson FP16. Pilot it as opt-in *after* the distance KF lands (which delivers most of the temporal-stability benefit on the existing depth, license-free and CPU-free).

The throughline: **ship the two pure-numpy heads (KF + TTC) on the depth you already have first** — they are additive, config-gated, fully testable, real-time on CPU, and unencumbered — then upgrade the depth *source* (metric DA-V2, then oVDA) once the licensing and statefulness questions are resolved.

## Calibration, accuracy & edge deployment

This section evaluates how to turn the platform's existing monocular-depth proximity feature (`src/depth/`, ONNX Depth-Anything/MiDaS writing `Detection.depth`, surfaced by `scripts/run_depth.py`) into **true metric range** to detected objects, and how to keep the detect-then-range pipeline **real-time on consumer webcams and edge (Jetson) hardware**. The two verified research cards converge on a single two-stage recipe: a fast, quantization-friendly detector followed by a small monocular metric-depth model, with the depth stage as the latency bottleneck and the metric-units conversion as the main correctness risk.

#### 1. The pipeline at a glance

The platform already does detect-then-sample: `detector.detect(frame)` produces `list[Detection]`, then `build_depth_estimator(...).estimate(frame)` produces an `HxW` depth map and `annotate_depth()` writes a per-object value into `Detection.depth` by calling `sample_depth()` (verified at `/home/user/watch/src/depth/base.py`). Today both the detector wrapper and the depth backend are wired for this; the research below is about (a) swapping in an edge-grade detector and (b) replacing the *relative* inverse-depth value with *absolute meters*.

| Stage | Recommended method | Role | One-line idea |
|---|---|---|---|
| Detect | **YOLO26** ([Ultralytics, 2025](https://docs.ultralytics.com/models/yolo26); benchmark study [arXiv 2509.25164, 2025](https://arxiv.org/abs/2509.25164)) | localize objects | NMS-free, DFL-free one-stage detector re-engineered for clean INT8/edge |
| Range | **Depth Anything V2 (Metric, Small)** ([Yang et al., 2024](https://github.com/DepthAnything/Depth-Anything-V2); [arXiv 2406.09414](https://arxiv.org/abs/2406.09414)) | distance per object | ViT-S + DPT monocular depth, metric fine-tune outputs meters directly |

#### 2. Detector: YOLO26 is the edge/quantization win

YOLO26 makes two structural changes that matter on edge hardware: it is **natively NMS-free** (one-to-one head, removing a CPU-side post-processing bottleneck and IoU-threshold tuning), and it **drops Distribution Focal Loss (DFL)** in favor of direct box regression, which is the key enabler for clean INT8 because DFL's custom decode op historically broke or slowed integer quantization ([arXiv 2509.25164, 2025](https://arxiv.org/abs/2509.25164)). The practical consequence — confirmed both in the paper's framing and the official Jetson table — is that INT8 exports retain nearly full-precision mAP.

**Accuracy (COCO val2017 mAP50-95, [Ultralytics docs, 2025](https://docs.ultralytics.com/models/yolo26)):**

| Model | mAP50-95 | Params | FLOPs |
|---|---|---|---|
| YOLO26n | 40.9 | 2.4 M | 5.4 B |
| YOLO26s | 48.6 | 9.5 M | 20.7 B |
| YOLO26m | 53.1 | 20.4 M | 68.2 B |
| YOLO26l | 55.0 | 24.8 M | 86.4 B |
| YOLO26x | 57.5 | 55.7 M | 193.9 B |

**Latency + the INT8 accuracy cost** are measured on the official Ultralytics Jetson benchmark (JetPack 6.1, Ultralytics 8.4.33), verified verbatim against [docs.ultralytics.com/guides/nvidia-jetson](https://docs.ultralytics.com/guides/nvidia-jetson):

|---|---|---|---|---|
| Orin Nano Super | YOLO26n | TensorRT FP16 | 4.57 | 0.4800 |
| Orin Nano Super | YOLO26n | TensorRT INT8 | 3.80 (~263 FPS) | 0.4490 |

INT8 costs roughly **3 mAP** on the nano (0.479 → 0.449) for a ~20% speedup — acceptable for a proximity use-case where localization, not fine classification, drives the alert. On **CPU** (Intel Xeon @2.00 GHz, ONNX Runtime) YOLO26n reportedly hits 40.9 mAP at 38.9 ms (~26 FPS) versus YOLOv8n's 37.3 mAP at 80.4 ms — higher accuracy and ~2× faster than v8n, making it a genuine CPU-webcam upgrade ([arXiv 2509.25164, 2025](https://arxiv.org/abs/2509.25164); [Ultralytics docs](https://docs.ultralytics.com/models/yolo26)).

**Verdict on real-time:** the detector is *not* the bottleneck. On any Jetson Orin it runs at hundreds of FPS in TensorRT INT8/FP16, and ~26 FPS on a desktop CPU. The headroom is entirely consumed by the depth stage (below).

#### 3. Range: Depth Anything V2 (Metric, Small) gives meters, but Small is the only edge-viable tier

The platform's current depth backend (`DepthAnythingV2` in `/home/user/watch/src/depth/onnx_estimator.py`, docstring "Larger value = nearer (inverse depth)") emits a **relative** map that `percentile_normalize()` squashes to [0,1]; `run_depth.py` then alerts when `det.depth >= 0.8` (verified line 102). The metric fine-tunes change this fundamentally: a DPT head **regresses depth in meters directly** — indoor on Hypersim (`max_depth = 20 m`), outdoor on Virtual KITTI 2 (`max_depth = 80 m`), so there are **separate indoor/outdoor checkpoints per scale** ([Depth Anything V2, 2024](https://github.com/DepthAnything/Depth-Anything-V2/blob/main/metric_depth/README.md)). Meters mean **smaller = nearer**, the *opposite* of the repo's current convention — this sign flip is the central correctness item.

**Latency (the gating constraint).** Real-time depth on edge is achievable *only* with the Small encoder and *only* at reduced resolution. The most directly applicable measurement is the [IRCVLab Jetson Orin sweep](https://github.com/IRCVLab/Depth-Anything-for-Jetson-Orin) (ViT-S, TensorRT 8.5.2, **Jetson Orin Dev Kit 8GB**), verified verbatim:

| Input size | Latency | FPS | Note |
|---|---|---|---|
| 308×308 | 23.5 ms | ~42.6 | real-time |
| 364×364 | 39.2 ms | ~25.5 | real-time |
| 406×406 | 47.7 ms | ~21 | real-time |
| 518×518 (repo default) | 98.0 ms | ~10.2 | borderline |

A separate Jetson benchmark of the *metric* DA-V2-Small (TensorRT BF16, batch-4 @480×300) reports 10.1 FPS on Orin NX, 19.9 FPS (50.16 ms) on AGX Orin, and 52.6 FPS on Jetson Thor ([arXiv 2603.01999](https://arxiv.org/abs/2603.01999), as cited in the card). **Two caveats** apply to the IRCVLab table and must be carried into the roadmap: it benchmarks the original **Depth Anything V1** ViT-S (not V2) and does not state FP16; however V1/V2-Small share the same ViT-S14 + DPT / 24.8 M-param architecture, so the figures are a fair throughput proxy (latency is encoder-bound) rather than an exact V2 metric measurement.

**Memory forces the Small tier regardless of anything else:** Base/Large variants do **not fit on Jetson Orin 8GB** (out-of-memory per IRCVLab), so Small is mandatory on that hardware. This happens to align with licensing (below).

**Accuracy.** Authors publish metric numbers chiefly for the **Large** encoder under the in-domain ZoeDepth protocol — e.g. NYUv2 (ViT-L) AbsRel 0.056 / RMSE 0.206 m versus ZoeDepth 0.077 / 0.282 ([arXiv 2406.09414, 2024](https://arxiv.org/abs/2406.09414)). These are **in-domain (fine-tuned on NYU/KITTI-style data), not zero-shot**, and not comparable to intrinsics-conditioned methods (Metric3D v2, UniDepth) scored without median alignment. **There is no published per-dataset metric table for the Small variant**, and an in-meters RMSE/AbsRel for *this* deployment must be measured on-platform against a known target — treat it as a calibration task, not a quoted number. The card deliberately excludes the widely-quoted "KITTI 0.036 / NYU 0.043 / ETH3D δ1 0.990" headline as unverifiable/mis-attributed; that exclusion is correct and those figures should not appear in the roadmap. (Note: arxiv.org and Hugging Face returned HTTP 403 during verification, so the Large/NYU figures rest on the official repo and corroborating sources rather than the primary PDF — confidence medium on the exact decimals.)

#### 4. The two non-obvious deployment rules

**Rule 1 — INT8 the detector, NOT the depth model.** TensorRT INT8 *regresses* speed on the ViT+DPT depth architecture because it breaks the fused FP16 attention kernels into plain matmuls. A direct NVIDIA benchmark of MiDaS V3 DPT-Hybrid on Jetson AGX Thor shows **FP16 at 5.8 ms vs INT8 at 10.3 ms (~1.8× slower)** ([NVIDIA Developer Forums, as cited in the card](https://forums.developer.nvidia.com/)). The recommended edge recipe is therefore **YOLO26 INT8 + DAV2-Small FP16/BF16** — mixed precision across the two stages.

**Rule 2 — depth dominates the frame budget; fix it with resolution and cadence, not a bigger GPU.** On an Orin Nano, a sequential pass is roughly YOLO26n INT8 (~3.8 ms) + DAV2-S FP16 @406px (~47.7 ms) ≈ **~52 ms/frame (~19 FPS)** — i.e. depth is >90% of the cost. The two cheap wins, both already supported by the repo's `input_size` config and tracker, are: (a) lower depth input from 518 → 384/406 (2–4× faster, no model change), and (b) run depth every Nth frame, reusing the last map for tracked boxes.

#### 5. Maturity & licensing (a real gate for commercial use)

| Component | Official weights | License | Implication |
|---|---|---|---|
| YOLO26 (n/s/m/l/x) | Yes ([Ultralytics](https://huggingface.co/Ultralytics/YOLO26)) | **AGPL-3.0** | Closed-source/commercial distribution needs a paid Ultralytics Enterprise license |
| DAV2-Small (incl. metric) | Yes ([repo](https://github.com/DepthAnything/Depth-Anything-V2)) | **Apache-2.0** | Commercially clean — and the only tier that fits 8GB Jetson |
| DAV2 Base/Large/Giant | Yes | **CC-BY-NC-4.0** | Non-commercial only; also OOM on 8GB Orin |
| onnxruntime / TensorRT / OpenVINO | n/a | permissive | exporter/ONNX path unaffected |

Both models are mature with official weights, docs, and Jetson/TensorRT guides; the platform already integrates DAV2 via `onnx_estimator.py`. **Maturity is not the risk — licensing (YOLO26 AGPL) and the meters/sign-flip conversion are.** The Apache-2.0 DAV2-Small is the only metric tier that is simultaneously commercially usable and 8GB-Jetson-feasible, so the constraints conveniently align on Small.

#### 6. Fit for this repo

Both changes are mostly config plus one small additive backend, and both respect the repo's golden rules (lazy heavy imports already in place; off-by-default; pure logic unit-testable with the model mocked).

- **Detector — essentially zero code.** `src/models/registry.py` line 39 already aliases `yolo26` to `YOLODetector` (verified), so `build_detector({"name": "yolo26n.pt"})` works today; the NMS-free head makes `iou_threshold` a harmless no-op. INT8/FP16/engine/OpenVINO export is already wired in `src/deployment/exporter.py`. Effort: **trivial** (swap weights + an export call + a config entry).
- **Depth — one small backend + a sign-flip.** Add a `depth_anything_metric` backend to `build_depth_estimator` in `/home/user/watch/src/depth/onnx_estimator.py`, reusing `preprocess`/`postprocess`/`OnnxDepthEstimator.estimate` almost verbatim. The real work is the **convention flip**: (1) the metric map is meters, larger=farther, so **bypass `percentile_normalize`** for this backend and add a units flag on `BaseDepthEstimator`/`Detection.depth`; (2) update the `base.py` docstring and change `scripts/run_depth.py` line 102 to alert when `det.depth <= proximity_threshold` (meters) instead of `>= 0.8`; (3) add an indoor-vs-outdoor checkpoint selector in config (Hypersim vs VKITTI). `sample_depth()` (20% bbox shrink + 3-sigma MAD + median) stays correct as-is and still needed to suppress background bleed. Effort: **~1 small PR**; unit-test the pure parts (units handling, meters-threshold comparison, `sample_depth` on a synthetic metric map) with a mocked session; validate the real forward and in-meters accuracy on a Jetson/torch box separately.

**Risks to carry into the roadmap:** (1) the **sign flip** — `run_depth.py` and any existing proximity tests must change together or alerts fire backwards; (2) **scale fragility** — DAV2 metric scale is baked in by Hypersim/VKITTI with no intrinsics input, so absolute meters can be biased on an out-of-domain webcam scene; pair with a one-time per-camera scale check against a known distance (or a focal-length estimate, e.g. GeoCalib) — unlike intrinsics-aware UniDepth/Metric3D; (3) **in-meters error is unmeasured** for this deployment and must be calibrated on-platform; (4) **temporal flicker** on video (Video Depth Anything addresses this separately); (5) verify the exact **INT8 TensorRT engine build** against your JetPack/TensorRT version before committing.

#### 7. Ranked recommendation for roadmap work

1. **Add the metric DAV2-Small depth backend + meters/sign-flip (highest value, low effort).** This is the single change that converts the existing relative proximity feature into *true range in meters* — the stated goal — and is a near-drop-in for `onnx_estimator.py`/`run_depth.py`. Apache-2.0, 8GB-Jetson-feasible. Ship with the indoor/outdoor checkpoint selector and the units flag; gate behind config so default behavior is unchanged.
2. **Adopt YOLO26 as the detector tier and wire its INT8/FP16 export (high value, trivial effort).** Registry already aliases it; gains higher mAP *and* lower latency than the current YOLOv8 default, on both Jetson (INT8) and CPU. The only blocker is AGPL licensing for commercial distribution — flag for product, not engineering.
3. **Implement the depth-cadence + resolution-throttle controls (high value, small effort).** Since depth is ~90% of the per-frame budget, expose depth `input_size` (target 384–406 on edge) and an "every-Nth-frame, reuse map for tracked boxes" path. This is what actually buys real-time (~19 FPS Orin Nano) without new models.
4. **Add an on-platform metric-calibration utility (medium value, medium effort).** A one-time per-camera scale check against a known distance to correct the baked-in scale bias, plus a small in-meters RMSE/AbsRel harness — because no Small-tier metric accuracy is published and accuracy is the report's priority.
5. **Enforce the mixed-precision rule in the exporter/config (low effort, prevents a regression).** Make INT8 detector + FP16/BF16 depth the documented edge default so nobody quantizes the DPT head and loses ~1.8×.

**Bottom line:** the verified evidence supports one concrete, real-time-feasible path — **YOLO26 (INT8 on Jetson / INT8 ONNX on CPU) + Depth Anything V2-Small metric (FP16/BF16)** — that drops cleanly into the existing detector factory and depth subsystem. The detector is effectively free; range quality and frame rate are governed entirely by the depth stage's resolution, cadence, and a per-camera scale calibration, with the meters/sign-flip and AGPL licensing as the two items to get right.

## Appendix A — method cards (one per domain, top method deep-read)

### Real-time object detectors — Ultralytics YOLO26 (NMS-free, edge-first real-time detector)

**Source:** [Ultralytics YOLO26: Unified Real-Time End-to-End Vision Models (Jocher, Qiu, Liu et al., Ultralytics, 2026); benchmark table from "YOLO26: Key Architectural Enhancements and Performance Benchmarking for Real-Time Object Detection" (arXiv 2509.25164); official model docs at docs.ultralytics.com/models/yolo26](https://arxiv.org/abs/2606.03748) (2026)

- **How it works:** YOLO26 is an anchor-free, single-stage detector that is natively end-to-end: it removes Non-Maximum Suppression (NMS) from inference. It is trained with a dual-head scheme that shares one backbone+neck: a classic one-to-many head feeds dense gradients to the backbone, while a one-to-one (top-1 matching) head learns to emit exactly one box per object, with a consistent matching metric aligning the two so the one-to-one head can be used alone at inference. Other changes vs the YOLOv8/v11 lineage: Distribution Focal Loss (DFL) is removed from the regression head (simpler, cleaner ONNX/TensorRT export and faster CPU post-processing), small-object accuracy is raised via STAL (Small-Target-Aware Label Assignment) plus ProgLoss (progressive/dynamic supervision), and training is stabilized with a new MuSGD optimizer. Net effect: a true end-to-end graph with no NMS threshold to tune and no variable per-frame NMS latency, which is the dominant CPU/edge win. Verified against the official paper (arXiv 2606.03748, Jocher/Qiu/Liu et al.) and Ultralytics docs; the per-variant benchmark table is from arXiv 2509.25164 and the official model page.
- **Inputs / outputs:** Input: a single RGB image/frame, default 640x640 (configurable; standard YOLO letterboxing). Output in end-to-end mode: a fixed, de-duplicated set of detections (class label, confidence, axis-aligned xyxy box) with no NMS post-step — one box per object directly. - **How it works:** A DINOv2 ViT-Small encoder feeds a DPT decoder head. ### Stereo & multi-view depth — ESMStereo (Enhanced ShuffleMixer Disparity Upsampling for Real-Time and Accurate Stereo Matching)

**Source:** [ESMStereo: Enhanced ShuffleMixer Disparity Upsampling for Real-Time and Accurate Stereo Matching (Tahmasebi, Huq, Meehan, McAfee — Atlantic Technological University / MISHE)](https://arxiv.org/abs/2506.21091) (2025)

- **How it works:** A lightweight, calibrated-stereo disparity network built around three stages: (1) a CNN feature-extraction encoder (MobileNetV2 for the small ESMStereo-S; EfficientNet-B2 for the medium/large ESMStereo-M/-L); (2) a deliberately SMALL-scale cost volume (groupwise-correlation, the "gwc" suffix) plus a lightweight 3D aggregation hourglass, which is cheap but normally loses fine detail; and (3) the paper's core contribution, the Enhanced ShuffleMixer (ESM) disparity-upsampling module that recovers that lost detail. ESM is a fast feature-guided upsampler: it extracts features from the coarse initial disparity, FUSES them with high-resolution image features, mixes them via channel shuffling + layer splitting (the ShuffleMixer operator borrowed from efficient super-resolution), then refines through a compact feature-guided hourglass to reconstruct a full-resolution disparity map. The ShuffleMixer gives a large receptive field / local contextual connectivity at low FLOPs, letting a tiny cost volume reach near-large-volume accuracy at real-time speed. Variants S/M/L (and gwc cost-volume option) let you trade D1 error for FPS. NOTE for this repo: ESMStereo outputs DISPARITY; converting to METRIC depth requires a calibrated, rectified stereo rig and Z = f*B/d (focal f in px, baseline B in m, disparity d). This is the key advantage over the repo's current monocular Depth-Anything/MiDaS path, which only yields relative inverse depth.
- **Inputs / outputs:** Input: a RECTIFIED stereo image pair (left + right), e.g. 380x1248 in the benchmarks. Output: a dense disparity map (per-pixel horizontal pixel shift), with an optional confidence map (the repo ships onnx_transformed_confidence.py / kitti_publisher_conf). Disparity is converted to metric depth externally via Z = f*B/disparity using the rig's calibrated focal length and baseline. Requires offline stereo calibration + rectification (intrinsics/extrinsics, cv2.stereoRectify / initUndistortRectifyMap) before inference; it does NOT do calibration itself.
- **Accuracy:** Scene Flow EPE (end-point-error, lower better), consistent across the official GitHub, arXiv, and ResearchGate: ESMStereo-S-gwc 1.10 px, ESMStereo-M-gwc 0.77 px, ESMStereo-L-gwc 0.53 px. KITTI-2015 (online leaderboard): ESMStereo-L-gwc D1-all = 1.82% (reported as ranking ~2nd on the KITTI-2015 leaderboard at submission). Reported model sizes: ~1.7M params (S-gwc) to ~6.6M params (L-gwc). Per-variant KITTI-2012 / per-variant S and M KITTI-2015 D1 figures were not verifiable in the sources I could open (arxiv.org PDF/HTML and the review mirror returned 403 via the egress proxy), so they are left out rather than guessed. (Caution: a survey draft cited S 10.8% / M 8.2% / L 5.5% D1-all — I could NOT verify those against any primary source; the only KITTI-2015 D1-all I could confirm is L-gwc = 1.82%.)
- **Latency / FPS:** Two self-consistent sets of official numbers exist; both are at 380x1248 with TensorRT FP16. (a) Paper abstract: the compact model reaches 116 FPS on a high-end GPU and 91 FPS on Jetson AGX Orin. (b) Official GitHub README (newer setup, JetPack 6.2.1 + TensorRT 10): ESMStereo-S ~105 FPS on an RTX 4070 Super; on Jetson AGX Orin 40GB, S ~130 FPS and L ~67 FPS. Per-variant raw inference time (Scene Flow table): S-gwc 8.6 ms, M-gwc 14 ms, L-gwc 26 ms. The 8.6 ms / 91 FPS figure for S-gwc on AGX Orin is the headline real-time-on-edge claim.

Software: PyTorch 2.5.1+cu118, Python 3.10, JetPack 6.2.1, TensorRT 10. ### Geometric monocular distance — Single-view ground-plane / IPM geometry (foot-point back-projection through a calibrated road-plane homography), per Ali et al. WACV 2020 — "Real-time vehicle distance estimation using single view geometry"

**Source:** [Real-time vehicle distance estimation using single view geometry (Ali, Hassan, Ali, Khan, Kazmi, Zaheer), WACV 2020, pp. 1111-1120](https://openaccess.thecvf.com/content_WACV_2020/papers/Ali_Real-time_vehicle_distance_estimation_using_single_view_geometry_WACV_2020_paper.pdf) (2020)

- **How it works:** Closed-form geometric ranging that piggybacks on an existing lane+vehicle detector. (1) Estimate the horizon line from the cross-ratios of detected lane boundaries (a projective invariant), so the vanishing line is recovered per-frame from image evidence rather than fixed. (2) Using a KNOWN real-world lane width plus the detected horizon, solve for the Inverse-Perspective-Mapping (IPM)/road-plane homography and the camera height — i.e. reconstruct the metric ground plane from a single image. (3) For each detected vehicle, take the bounding-box foot point and back-project it as a ray from the camera centre, intersecting it with the reconstructed road plane to read off metric (X, Z) ground position, hence range. A per-frame "virtual horizon" is recovered from lane/scene geometry so that pitch changes from vehicle motion and road slope are absorbed instead of being baked into a one-time calibration. Net effect: range comes from a few matrix operations on the detector's bbox, with no learned depth network. Evaluated against lidar ground truth and compared with radar and a deep monocular-depth baseline (DORN).
- **Inputs / outputs:** Inputs: a single RGB frame; per-frame detections of lane boundaries (to recover the horizon/virtual horizon via cross-ratios) and of vehicles (bbox, whose bottom-centre foot point is the measured quantity); ONE metric scale prior — a known lane width — from which camera height and the IPM homography are derived; camera intrinsics (focal length / principal point). Outputs: per-detection metric position on the road plane (longitudinal range, and lateral offset), i.e. an absolute distance in metres to each vehicle. The recovered horizon, camera height and IPM homography are intermediate outputs reusable for BEV mapping.
- **Accuracy:** Reported consistent RMSE of 6.10-7.31 m across three public driving datasets — KITTI, nuScenes, and Lyft Level 5 — at automotive ranges (up to ~100 m), with lidar as ground truth. For cross-camera robustness the authors note a deep baseline (DORN trained on KITTI) degrades to RMSE 12.23 m when deployed on a camera with different settings, whereas this geometric method does not require per-camera retraining. Error grows with true range and is dominated by deviation of the real scene from the calibrated flat road plane (pitch/roll, slope) and by the lane-width/horizon assumptions.
- **Latency / FPS:** The distance-estimation computation itself takes ~2.31e-05 s (about 23 microseconds) per detection on a Core i7 CPU — effectively free relative to the detector. There is no dedicated depth-network forward pass; total system FPS is therefore bounded by the upstream lane + vehicle detectors, not by the ranging step. No standalone full-pipeline FPS or GPU timing is isolated in the paper beyond the per-detection CPU figure.
- **Hardware:** Per-detection ranging timing measured on a Core i7 CPU (the geometric step is CPU-only and GPU-free). End-to-end speed depends on whatever runs the lane and vehicle detectors; the paper frames the approach as a lightweight add-on to an existing ADAS.
- **License:** Not verified. No official public code repository or license was found via search; the paper (WACV 2020, work done at KeepTruckin Inc.) does not advertise released weights/code. The geometry itself is a standard projective construction and is not patent/license-encumbered to reimplement, but no first-party implementation license is confirmed — left blank rather than guessed.
- **Fit for this repo:** Strong fit and the best webcam/edge match in this family. It maps almost 1:1 onto existing primitives in src/multicam/geometry.py: build_homography() (image<->ground correspondences), ground_point() (already returns the bbox bottom-centre foot point), and project_point() (perspective back-projection). Recommended shape: add a single-camera GroundPlaneRanger in a new src/depth/ground_plane.py (or src/geometry/) implementing BaseDepthEstimator-style output, that takes a calibrated homography + camera height/intrinsics and writes an ABSOLUTE metric value into Detection.depth — the same field src/depth/annotate_depth() populates today, so run_depth.py proximity alerts work unchanged but in true metres. Gate it in configs/default.yaml behind a depth.backend: "ground_plane" option alongside the existing depth_anything/midas factory, keeping defaults unchanged (additive + config-gated per CLAUDE.md). Effort: LOW for the core (a calibration block + a few matrix ops, fully unit-testable with pure numpy/cv2, no torch — fits the slim-deps invariant); MODERATE if you also automate the lane-cross-ratio horizon recovery rather than requiring a one-time static calibration. Risks/mitigations: needs a per-camera calibration (a known ground reference or lane width); sensitive to pitch/roll and non-flat ground — pair the foot point with the existing tracker for temporal smoothing of foot-point jitter, and consider the WACV virtual-horizon idea (or a later closed-form homography error-correction layer) to de-bias far-range estimates.
- **Limitations:** (1) Flat-ground assumption: range is read off a single reconstructed road plane, so slopes, curbs, speed bumps, and uneven terrain inject error that grows with distance. (2) Pitch/roll sensitivity: camera attitude deviation from calibration is the dominant error source; the per-frame virtual horizon mitigates but does not eliminate it. (3) Requires a metric scale prior (known lane width) and visible lane boundaries to recover the horizon — degrades where lane markings are absent/occluded (intersections, parking lots, generic indoor/webcam scenes outside the road domain). (4) Depends on a tight, well-localized bbox foot point; occlusion or bbox jitter at the object's base maps directly to range error. (5) Accuracy is coarser than learned metric-depth methods (RMSE 6-7 m at driving ranges), so it suits relative ranking / proximity alerts more than centimetre-grade measurement. (6) Calibration is camera-specific and assumes the object stands on the ground plane (fails for non-ground objects). (7) No official released code/weights confirmed, so it must be reimplemented.
- **Fact-check:** source verified: yes; numbers plausible: yes; confidence: high. Paper is REAL and correctly attributed. Verified via WACV 2020 open-access page, IEEE Xplore (doc 9093634), and Semantic Scholar: title, all six authors (Ahmed Ali, Ali Hassan, Afsheen Rafaqat Ali, Hussam Ullah Khan, Wajahat Kazmi, Aamer Zaheer), venue/year, URL, and KeepTruckin Inc. funding all match. ### Monocular 3D detection & range — Detector + monocular-depth fusion for per-object range (YOLO/RT-DETR 2D box + Depth-Anything-V2 depth sampled inside the box -> Detection.depth), i.e. the Dist-YOLO "detect + distance in one pass" pattern realized as two cooperating models

**Source:** [Dist-YOLO: Fast Object Detection with Distance Estimation (Vajgl, Hurtik, Nejezchleba), Applied Sciences 12(3):1354, 2022 — official code MIT at gitlab.com/EnginCZ/yolo-with-distance; depth backend: Depth Anything V2 (NeurIPS 2024), github.com/DepthAnything/Depth-Anything-V2](https://www.mdpi.com/2076-3417/12/3/1354) (2022)

- **How it works:** Run a fast 2D detector (YOLO/RT-DETR) and a monocular depth network on the same frame, then read the depth values inside each 2D box to assign a range to every detected object. Dist-YOLO (Vajgl/Hurtik/Nejezchleba, Applied Sciences 12(3):1354, 2022) formalizes the single-network version: it extends a YOLOv3-class detector's prediction vector with an extra distance term, shares the backbone weights with the box regressor, and adds a distance loss; distance can be encoded class-agnostic (smaller vectors, better results) or class-aware, and the box and distance subtasks are shown to be in synergy. This repo instead realizes the same goal as TWO cooperating models, which is what src/depth already implements: scripts/run_depth.py calls detector.detect(frame), then depth_estimator.estimate(frame) (ONNX Depth-Anything-V2 or MiDaS-small via build_depth_estimator), normalizes with percentile_normalize, and annotate_depth -> sample_depth shrinks each box toward its center (default shrink=0.2), applies a 3-sigma MAD outlier rejection, and takes the median, writing it to Detection.depth. Note: today the value stored is RELATIVE inverse depth in [0,1] (larger = nearer), not metres. The literature roadmap upgrade is to swap in a metric Depth-Anything-V2 checkpoint (or a Dist-YOLO-style trained head) so Detection.depth becomes an actual distance.
- **Inputs / outputs:** Input: a single monocular RGB frame from webcam/IP/RTSP/file (no stereo, no calibrated rig). ### Collision / time-to-contact — Closed-form bounding-box scale-change / optical-flow looming time-to-collision (tracker-driven)

**Source:** [Manglik, Weng, Ohn-Bar, Kitani - Forecasting Time-to-Collision from Monocular Video: Feasibility, Dataset, and Challenges (IROS 2019); canonical indoor monocular TTC dataset/baseline. Corroborating real-time benchmark: Toettrup et al. - A Real-Time Method for Time-to-Collision Estimation from Aerial Images (J. Imaging 2022). Closed-form definition: criticality-metrics TTC reference.](https://github.com/aashi7/NearCollision) (2019)

- **How it works:** A learning-free, per-track estimator that derives time-to-collision (TTC) from how fast a tracked object grows in the image. Two equivalent formulations: (1) Bounding-box scale-change. Take the object's image size (width w, height h, or sqrt(area)) at two times and form the relative scale S = size(t2)/size(t1) over interval dt. Under a constant relative-velocity pinhole model, image size is inversely proportional to range Z, so dS/dt encodes the closing rate and TTC = dt * S/(S-1) ~= scale / (rate-of-change-of-scale). Equivalently, TTC = -Z/(dZ/dt) = -dl/dv (the standard constant-velocity longitudinal form on criticality-metrics), which the repo can evaluate directly from the time-derivative of Detection.depth instead of bbox size. S>1 (box expanding / looming) gives a finite positive TTC; S<=1 (receding) gives no collision. (2) Optical-flow looming. Compute sparse/dense flow, locate the focus of expansion (FOE), and measure flow-field divergence (relative rate of expansion) about it: the optical variable tau = Z/W is the inverse relative expansion rate, and divergence is inversely proportional to TTC (tau ~= 1/divergence). Both are a handful of arithmetic ops per object once detection+tracking (or flow) is already running; robust deployment adds temporal smoothing (EMA / 1-D Kalman on size or 1/Z), a minimum-scale-change deadband to reject jitter, and a horizon gate (alert only when TTC < T_warn). No metric camera calibration is required for the bbox-ratio form because the unknown true object size cancels in the ratio; calibration only matters if you want absolute range rather than TTC.
- **Inputs / outputs:** Inputs (bbox form): a stable per-track sequence of bounding boxes with track_id and timestamps - exactly what src/tracking/EnhancedTracker + TrackHistory and src/analytics/temporal.TemporalBuffer already store (positions over time; box w/h available from Detection.width/height). Optionally Detection.depth (already written by src/depth) to use the depth-derivative form TTC = -Z/(dZ/dt). Inputs (flow form): consecutive grayscale frames for optical flow (cv2.calcOpticalFlowFarneback / Lucas-Kanade) plus an FOE estimate. Outputs: per-track scalar TTC in seconds (or +inf when receding/static), a derived closing-rate, and a boolean looming/collision-risk flag when TTC < threshold - emitted into the existing AlertManager as an alert_type like collision/ttc, mirroring the proximity rule in scripts/run_depth.py.
- **Accuracy:** No single canonical accuracy number for the closed-form method itself: quality is bounded by detector/tracker box stability and frame rate, not by the (exact) TTC arithmetic. Family-level published benchmarks for real-time monocular-vision TTC: Toettrup et al., "A Real-Time Method for Time-to-Collision Estimation from Aerial Images" (J. Imaging 2022) report mean error 0.358 s with std 0.114 s in a worst-case scenario, derived from scale changes of (rotated) bounding boxes tracked across frames - though on top-view aerial vessels, which bounds the family rather than the forward-looming variant. The canonical indoor monocular benchmark/dataset to evaluate against is Manglik et al. (IROS 2019, NearCollision): 13,000+ indoor video segments of a person approaching, with stereo-camera-derived ground-truth time-to-near-collision; that paper notes physics/scale-change baselines are brittle to bbox noise (small depth/box errors masquerade as large velocity changes) and motivates a learned alternative - i.e. the closed-form head is a strong, cheap baseline but not the accuracy ceiling. Reported numbers above come from those papers' own benchmarks, not from this repo; no model forward was run here.
- **Latency / FPS:** Effectively free relative to detection: a few arithmetic operations per track per frame on CPU (the bbox-ratio form is O(num_tracks)); the real-time ceiling is set by the existing YOLO/RT-DETR detector + tracker, not by the TTC math. The optical-flow looming variant adds the cost of dense/sparse flow per frame (the dominant cost in that path). Family reference for an end-to-end real-time pipeline: Toettrup et al. report 123.5 FPS at 1024x2048 with a lightweight segmentation CNN suitable for embedded devices (that figure includes their detection/segmentation, so it is an upper bound on the whole pipeline, not the TTC step alone).
- **Hardware:** CPU / any edge device (Jetson, laptop webcam); the TTC computation inherits the detector/tracker hardware budget and adds negligible load. The 123.5 FPS @ 1024x2048 reference figure (Toettrup et al.) was reported for a lightweight segmentation CNN described as suitable for embedded devices with low processing/memory specs; the paper does not name a specific CPU/Jetson SKU for that number.

- **Limitations:** (1) Box-noise amplification: TTC = S/(S-1) blows up as S->1, so small jitter/oscillation in YOLO boxes produces huge TTC variance - the dominant failure mode noted by Manglik et al.; requires EMA/Kalman smoothing on size (or 1/Z) and a minimum-scale-change deadband. (4) Centroid vs FOE: bbox scale-change assumes the object is roughly on the collision axis; off-axis objects need the FOE/divergence form or a lateral-motion correction. (5) Low frame rate / motion blur on fast approaches gives too few clean samples - the exact gap EvTTC argues event cameras fill. (6) TTC, not range: the bbox-ratio form yields time-to-contact but not absolute distance; absolute range still needs the existing monocular depth (Detection.depth) or a calibrated pinhole/known-size estimate. (7) Optical-flow variant is sensitive to textureless regions, illumination change, and ego-rotation, and its per-frame flow cost can dominate the budget on edge hardware.

### Calibration & distance accuracy — Depth Anything V2 (Metric, Small) via ONNX/TensorRT

**Source:** [Depth Anything V2 (NeurIPS 2024) — Yang, Kang, Huang, Zhao, Xu, Feng, Zhao; official repo + metric_depth fine-tuning (Hypersim/Virtual KITTI) and per-variant license](https://github.com/DepthAnything/Depth-Anything-V2) (2024)

### Edge real-time deployment — YOLO26 (TensorRT INT8/FP16 on Jetson; ONNX/OpenVINO INT8 on CPU) for detection + Depth Anything V2-Small (TensorRT FP16, metric fine-tune) for per-object range

**Source:** [YOLO26: Key Architectural Enhancements and Performance Benchmarking for Real-Time Object Detection (arXiv 2509.25164, 2025) + Ultralytics YOLO26 NVIDIA Jetson benchmarks; Depth Anything V2 (arXiv 2406.09414, NeurIPS 2024) + official metric_depth checkpoints](https://arxiv.org/abs/2509.25164) (2025)

- **How it works:** Two-stage, single-camera proximity pipeline: a fast detector localizes objects, then a monocular depth model gives the distance to each detection.

DETECTOR - YOLO26 (Ultralytics, Sept 2025). A one-stage YOLO re-engineered specifically for edge/quantization. Two structural changes matter: (1) it is natively NMS-free (end-to-end one-to-one head; no Non-Maximum-Suppression post-processing), which removes a CPU-side latency bottleneck and the threshold tuning that came with it; (2) it drops Distribution Focal Loss (DFL), replacing distributional box decoding with a lighter, hardware-friendly direct regression. Removing DFL is the key enabler for clean INT8 because DFL's custom decode op historically broke or slowed integer quantization. Training-side additions (not inference cost): ProgLoss (progressive loss balancing), STAL (Small-Target-Aware Label assignment, better small-object recall on edge inputs), and a new MuSGD optimizer. Net effect: INT8/FP16 exports keep nearly full-precision mAP and run fast on Jetson and CPU. (arXiv 2509.25164; Ultralytics docs.)

DEPTH/RANGE - Depth Anything V2-Small (NeurIPS 2024). A ViT-S encoder + DPT decoder monocular depth foundation model (24.8M params). The base model outputs relative inverse depth; the official Metric fine-tunes (indoor on Hypersim, max 20 m; outdoor on Virtual KITTI 2, max 80 m) output absolute depth in METERS directly (the repo confirms 'depth map in meters in numpy'). Per-object range = sample the depth map inside each detection's bbox and take a robust statistic (median). The repo's src/depth/base.py sample_depth() already implements exactly this (shrink bbox 20% toward center to avoid background leak, 3-sigma MAD outlier rejection, then median).

DEPLOYMENT SPLIT (verified, important): detectors go INT8 (fastest, ~3 mAP cost) but depth should stay FP16/BF16. TensorRT INT8 on the ViT+DPT depth architecture regresses speed: an NVIDIA forum benchmark of MiDaS V3 DPT-Hybrid on Jetson AGX Thor shows FP16 at 5.8 ms vs INT8 at 10.3 ms (INT8 ~1.8x SLOWER), because INT8 breaks the fused FP16 attention kernels into plain matmuls. So the recommended edge recipe is YOLO26 INT8 (Jetson) or INT8 ONNX/OpenVINO (CPU) + DAV2-Small FP16/BF16.
- **Inputs / outputs:** INPUTS: single BGR frame from the existing VideoStream (webcam / IP / RTSP / file). Detector input 640x640 (default); depth input is resizable (DAV2 supports 308/364/406/518) - the repo's OnnxDepthEstimator.preprocess already resizes, normalizes with ImageNet mean/std, and produces NCHW float32.

OUTPUTS: detector -> list[Detection] (bbox xyxy, confidence, class_id/name, optional track_id) via YOLODetector._results_to_detections. ### Temporal distance & tracking — Online Video Depth Anything (oVDA) as the temporally-consistent depth source, paired with a per-track 1D constant-velocity Kalman filter on each object's distance

**Source:** [Online Video Depth Anything: Temporally-Consistent Depth Prediction with Low Memory Consumption (3DV 2026); paired with the SORT/BoT-SORT constant-velocity Kalman-filter motion model applied to per-track distance](https://arxiv.org/abs/2510.09182 (oVDA paper); https://github.com/FriedFeid/OnlineVideoDepthAnything (official code + weights + license); https://openreview.net/forum?id=26S2CwuAf4 (3DV 2026 OpenReview); https://github.com/DepthAnything/Video-Depth-Anything (offline predecessor VDA, CVPR 2025); https://arxiv.org/pdf/2206.14651 (BoT-SORT, constant-velocity Kalman motion model reused for the per-track distance filter)) (2026)

- **How it works:** Two complementary pieces. (A) oVDA (Feiden, Kuchler, Zavadski, Savchynskyy, Rother; Heidelberg University; 3DV 2026; arXiv 2510.09182) turns the offline Video-Depth-Anything (VDA) into a STREAMING video-depth model by borrowing two tricks from LLM inference: (1) it caches a sliding window of past latent features so each new frame is fused with recent temporal context instead of re-running a fresh batch, and (2) it MASKS frames during training so the network learns to predict the current frame from cached/partial context, matching the online regime. A frozen DINOv2 ViT-S encoder feeds a lightweight "Motion Module"; a new scale-and-shift consistency loss enforces temporal stability. The result is flicker-free depth produced frame-by-frame at low latency and low VRAM, vs. the original VDA which needs a full 32-frame window (1x32x518x518) per forward and is therefore offline/windowed (>1 s latency at 30 fps). oVDA outputs scale-and-shift-invariant (relative, inverse-style) depth at input size 518; the repo already consumes exactly this kind of map. (B) On top of the depth stream, a tiny per-track 1D constant-velocity Kalman filter runs on the SCALAR distance of each tracked object: state [D, v]^T (range and its rate), constant-velocity transition, measurement = the per-detection depth sampled inside the bbox (the repo's sample_depth median+MAD), updated every frame a detection is present and predicted through dropouts with growing covariance. This yields a smoothed range, a free closing-speed estimate, and graceful gap-filling. It is pure numpy (microseconds/track on CPU) and is additive to the bbox KF that ByteTrack/BoT-SORT already run (those Kalman-filter the box, never the distance). Net effect: oVDA removes single-frame depth flicker at the source; the distance KF adds cross-frame smoothing + velocity per object.
- **Inputs / outputs:** oVDA input: a live monocular RGB stream (webcam/IP/RTSP/file), default inference size 518 (model resizes internally; the repo's preprocess already does BGR->RGB, resize, ImageNet normalize, NCHW). Output: a per-frame HxW float depth map, scale-and-shift-invariant (RELATIVE depth, not metric), temporally consistent across frames; same shape contract as today's OnnxDepthEstimator.estimate(). Crucial difference: oVDA is STATEFUL - it carries cached latent features across calls, so the estimator object must persist between frames and be reset on stream change/seek (unlike the current stateless per-frame ONNX call). Distance KF input per track: the scalar depth sampled in the detection bbox (existing sample_depth output) keyed by Detection.track_id. KF output per track: smoothed_distance (relative range) and velocity (range rate, sign = approaching/receding); converts to metric m and m/s ONLY with external scale calibration. These feed Detection.depth and run_depth.py proximity alerts.
- **Accuracy:** oVDA zero-shot video depth (scale-and-shift aligned to the FIRST frame, then applied to the whole clip - i.e. measuring temporal consistency, not metric accuracy), KITTI / Bonn / Sintel: AbsRel 0.140 / 0.118 / 0.380; delta1 0.809 / 0.871 / 0.548 (oVDA-small, ViT-S). Compared against ChronoDepth, NVDS, CUT3R, single-frame Depth-Anything-V2 (DAv2-s, DAv2-l) and FlashDepth-s; paper reports oVDA is the temporally most consistent online method and best on accuracy + VRAM among online methods, on par with FlashDepth on fine detail. Context for the offline predecessor VDA (CVPR 2025): delta1 0.910 (KITTI) / 0.908 (NYUv2) for the Large model - higher, but it needs the 32-frame window and is not real-time. The per-track 1D distance KF is a standard CV Kalman filter; its distance specialization is NOT independently benchmarked (no published AbsRel/MOTA for "KF-on-range" specifically) - leave its numeric accuracy blank and validate empirically in-repo. NOTE: arxiv.org and openaccess.thecvf.com PDFs were egress-blocked (HTTP 403) by the org proxy; oVDA numbers are from the paper's results as surfaced via search + the official GitHub repo, so treat exact per-dataset figures as to-be-reconfirmed against the PDF.
- **Latency / FPS:** oVDA (official, ViT-S small): 42 FPS on NVIDIA A100; on edge NVIDIA Jetson Orin NX 9 FPS at FP32 and 20 FPS at FP16 with no major accuracy drop (stability across precisions attributed to the frozen encoder + numerically benign Motion Module). Two cache sizes: oVDA-small-c16 (longer cache, slightly slower, better consistency) and oVDA-small-c8 (smaller cache). For contrast, offline VDA on A100 FP16 (input 1x32x518x518): 7.5 ms/window (Small), 14 ms/window (Large) - but the mandatory 32-frame window makes it offline (>1 s latency at 30 fps). The 1D distance KF adds negligible overhead (scalar update, microseconds/track on CPU); end-to-end FPS is bounded by detector + depth, not the filter. On a pure CPU consumer webcam, oVDA (ViT-S) will be the bottleneck and likely sub-real-time without GPU/TensorRT; the realistic real-time target is Jetson Orin NX FP16 (~20 FPS) or a desktop GPU.
- **Hardware:** oVDA measured on NVIDIA A100 (42 FPS) and NVIDIA Jetson Orin NX (9 FPS FP32 / 20 FPS FP16). Ships ONNX + TensorRT compilation scripts (src/build_ONNX/compile_onnx.py, compile_tensorrt.py) for edge deployment. Distance KF: any CPU (consumer webcam / Jetson).

## Appendix C — completeness critique

_Overall confidence (critic agent): **medium**_

**Gaps**

- FRESHNESS — Depth Anything 3 supersedes the metric-depth card's anchor. Every depth-centric card (mono-metric-depth, detection-depth-fusion, mono-3d-detection, calibration-accuracy, edge-deployment) is built on Depth Anything V2 (2024). DA3 shipped Nov 2025 (ByteDance-Seed, arXiv 2511.10647, ICLR 2026, Apache-2.0, official repo github.com/ByteDance-Seed/Depth-Anything-3) with a dedicated metric series (DA3METRIC-LARGE; meters = focal * net_output / 300), reportedly beats DA2 in monocular depth, plain DINO backbone (clean ONNX export), and already has a community ROS2+TensorRT node (ika-rwth-aachen/ros2-depth-anything-v3-trt). The roadmap should not commit DAv2 engineering without at least a head-to-head vs DA3-Base/Small. Per-variant latency/params and a no-intrinsics metric mode for DA3 were NOT verified and need a deep pass.
- WITHIN-DOMAIN COVERAGE COLLAPSE — the metric-depth survey scope named Metric3D/Metric3Dv2, UniDepth/UniDepthV2, ZoeDepth, and Depth Pro (see DOMAINS[mono-metric-depth] in .claude/workflows/distance-detection-research.js), but the delivered card covers ONLY Depth Anything V2. None of these alternatives got a card, numbers, or a fit note. This matters because UniDepthV2 (arXiv 2502.20110) and Metric3D V2 (TPAMI 2024, arXiv 2404.15506) produce METRIC depth WITHOUT known camera intrinsics / with a canonical-camera transform — directly addressing the uncalibrated-webcam->meters problem the calibration-accuracy card struggles with. Depth Pro (arXiv 2410.02073) gives sharp metric depth + focal length in <1s. These are the strongest answers to 'meters on an unknown webcam' and are essentially unaddressed.
- UNCOVERED DOMAIN — self-supervised / lightweight monocular depth for edge. Not in the DOMAINS list at all, yet it is arguably the single best fit for 'real-time on consumer webcam/edge' and for this repo's slim-deps ethos. Concrete 2025 methods with verified edge numbers: RTS-Mono (arXiv 2511.14107, ~3M params, ~20 FPS on Raspberry Pi 4, 49 FPS Jetson Orin), BoRe-Depth (arXiv 2511.04388, 8.7M params, 50.7 FPS Jetson Orin), plus Lite-Mono/MonoLENS/EdgeMono/RTIA-Mono. These train from video without labels (no GT depth), which suits a webcam R&D platform, and are 1-2 orders of magnitude smaller than DAv2-Large. Note: they are RELATIVE/scale-ambiguous (KITTI-tuned), so they pair with the scale-recovery gap below.
- UNCOVERED DOMAIN — active depth sensors / RGB-D (the lowest-effort, highest-accuracy distance path for an 'edge platform'). The entire survey is passive monocular/stereo and never covers reading depth directly from an Intel RealSense D435/D455, StereoLabs ZED 2, Luxonis OAK-D, or miniature ToF arrays. This is the dominant real-world deployment choice: hardware-aligned depth, RealSense D435 reports <1 cm error at <1 m (arXiv 2501.07421 four-camera comparison; multiple YOLO+RealSense pipelines exist). For this repo it is the cleanest fit of all — a pyrealsense2/depthai source feeding the EXISTING sample_depth()/annotate_depth() path in src/depth/base.py with zero new ML. The detection-depth-fusion card's scope explicitly mentioned 'RGB-D / frustum fusion' but the delivered card was DAv2 again, so RGB-D hardware was scoped-in but never delivered.
- WITHIN-DOMAIN GAP — classical OpenCV stereo (StereoSGBM) was skipped in favor of a learned net. The stereo-depth card delivered ESMStereo (good), and the survey scope named RAFT-Stereo/IGEV-Stereo/FoundationStereo, but nobody covered the zero-dependency cv2.StereoSGBM + stereoRectify + reprojectImageTo3D / Q-matrix path (Z = f*B/d). For an OpenCV-centric repo with a 'lazy heavy imports / slim deps' golden rule, two-commodity-webcam classical stereo is the most natural metric-distance baseline (pure cv2+numpy, unit-testable, no torch) and is absent. Known caveat to capture: error grows quadratically with distance and SGBM is resolution-bound — both relevant to real-time feasibility.
- UNCOVERED DOMAIN — camera-radar (and LiDAR) fusion for robust long-range range + direct velocity. Given the platform claims 'edge hardware,' mmWave-radar fusion is a fair deeper-pass candidate and was omitted entirely. Recent: RaCFormer (arXiv 2412.12725, 64.9 mAP/70.2 NDS nuScenes), RICCARDO (arXiv 2504.09086), CRN (ICCV 2023). Radar gives metric range + radial velocity directly, which would also strengthen the collision/TTC story (closing-velocity without depth differentiation). Lower priority for pure-webcam use, but should be explicitly acknowledged rather than silently dropped.
- METHODOLOGY GAP — no metric scale-recovery / calibration recipe or distance-accuracy benchmark plan. The calibration-accuracy card is itself another DAv2 instance, not a calibration METHOD. Missing: (a) how to turn relative/inverse depth into meters (known-object-height priors, fixed camera-height/IPM, ARUCO/ChARUCO ground truth) and (b) an evaluation protocol. Concrete anchors exist: 'Camera Height Doesn't Change' (arXiv 2312.04530) for scale via camera height; 'Language as Prior, Vision as Calibration' (arXiv 2601.01457, 2026) for metric-scale recovery; the wildlife metric-depth benchmark (arXiv 2510.04723: DAv2 MAE 0.454 m vs ChArUco GT) as a ready evaluation template; and 'How to Evaluate Monocular Depth Estimation?' (arXiv 2510.19814) for eval pitfalls. Without this, none of the depth cards can be validated for distance accuracy on this repo's own footage.
- DETECTOR BREADTH — realtime-detectors and edge-deployment are YOLO26-only; transformer detectors (which the repo already supports via rtdetr_wrapper.py) were not surveyed for 2026. Missing and clean fits to src/models/registry.py: RF-DETR (Roboflow, arXiv 2511.09554, ICLR 2026 — first real-time detector >60 mAP COCO at <=40 ms, DINOv2 backbone); RT-DETRv4 (Nov 2025, vision-foundation-model boosted); D-FINE. The card is also internally inconsistent on sourcing (it cites third-party survey arXiv 2509.25164 as if primary; the official YOLO26 paper is arXiv 2606.03748). A one-detector roadmap under-serves the 'object detection' half of the brief.
- TEMPORAL/FRESHNESS — newer 2026 streaming-depth work beyond the single tracked method. The track-distance card covers Online Video Depth Anything (3DV 2026, non-commercial license — a real adoption blocker it correctly flags). It misses AsyncMDE (arXiv 2603.10438, 2026, real-time monocular depth via asynchronous spatial memory), a candidate that could give temporally consistent per-frame depth without the NC license. Worth a deeper pass given the platform's real-time streaming focus.
- UNADDRESSED CROSS-CUTTING CONCERNS for roadmap decisions: (1) LICENSING — DAv2 Base/Large/Giant and oVDA are CC-BY-NC / non-commercial; an explicit 'commercially-usable distance stack' shortlist (Apache-2.0 DA3 / RTS-Mono / classical stereo / RGB-D SDKs) is not assembled. (2) NO FULL-PIPELINE END-TO-END BUDGET — almost every latency figure is network-only or backbone-only (e.g. ESMStereo Jetson FPS excludes pre/post + H2D/D2H per the verifier; Fast-SCNN 123 FPS is segmentation-only); detector + depth + tracker running CONCURRENTLY on one consumer GPU/edge SoC is never budgeted, which is the number that actually decides feasibility. (3) FAILURE MODES of monocular distance (transparent/reflective/low-texture surfaces, domain shift) are not characterized — NTIRE 2025 specular/transparent depth challenge (arXiv 2506.05815) is a ready reference.
